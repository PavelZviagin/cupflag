# WRITEUP - LVL2

## Запуск

```bash
python run_lvl2.py --duration 300
```

Флаги - в stdout и `flags_lvl2.jsonl`. На старте раннер поднимает пул сессий
(строка `pool: …/… sessions authenticated`).

## Архитектура решения

Код - в `cupflag/lvl2/` (`config.py` + `pool.py` + `runner.py`), общий каркас - в
`cupflag/common/`.

**Протокол.** То же окно захвата, что на LVL1, плюс капча Cloudflare Turnstile и
rate-limit. Каждый `POST /v1/capture` отправляет `captcha_token` и заголовок
`X-Request-Id`; ответы - `closed`, `ok`+`flag` или
`rate_limited`+`retry_after_ms`.

**Капча не нужна.** Сервер проверяет только наличие непустого `captcha_token` и
**не вызывает siteverify**: пустой/отсутствующий токен -> `400 captcha_required`,
любая непустая строка принимается. Шлём константный dummy-токен - без браузера.

**Настоящий гейт - rate-limit, пер-сессионный.** Сессия после запроса блокируется
на ~13 c, но свежая сессия независима (лимит привязан к `sid`, не к IP).

**Стратегия - RTT-залп из пула на границе секунды.** Та же идея, что на LVL1, но
каждый выстрел - из **своей** сессии:

- держим пул из `pool_size` (по умолчанию 48) предварительно залогиненных сессий;
- на каждой границе берём `shots` (по умолчанию 3) сессий, не на кулдауне, и шлём
  залп, таймированный по измеренному RTT (центр = `target_arrival − RTT/2`);
- каждая сессия используется не чаще раза в ~13 c -> лимит не срабатывает; пул
  мал, поэтому соединения переиспользуются и остаются тёплыми;
- фоновая задача **пополняет пул** (перелогинивает умершие сессии);
- кулдаун после флага, как на LVL1.

Пул нужен именно из-за пер-сессионного лимита; маленький залп (а не сплошной
поток) не вызывает у Cloudflare троттлинг соединений. Итог: ~6 флагов/мин,
**0 rate-limit**, ~20 запросов на флаг.

## Найденные уязвимости

1. **Turnstile не проверяется на сервере** - принимается любой непустой
   `captcha_token`; siteverify не вызывается. Защита косметическая, обходится
   константной строкой.
2. **Rate-limit пер-сессионный, а не пер-IP.** Вместе с бесплатным созданием
   аккаунтов это позволяет одному хосту полностью обойти лимит пулом одноразовых
   сессий.
3. **Наследуется с LVL1:** пароль `md5(username)` и глобальные флаги без привязки
   к сессии.
